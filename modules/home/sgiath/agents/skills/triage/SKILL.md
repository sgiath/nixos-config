# Triage

Move Backlog.md tasks through a small state machine of triage labels while preserving compact native Backlog statuses for the board.

Every task description, implementation note, plan, or final summary written during triage **must** start with this disclaimer:

```
> *This was generated by AI during triage.*
```

## Reference docs

- [agent-brief.md](reference/agent-brief.md) — how to write durable agent briefs
- [out-of-scope.md](reference/out-of-scope.md) — how the `.out-of-scope/` knowledge base works

## Roles

Two **category** roles:

- `bug` — something is broken
- `enhancement` — new feature or improvement

Five **state** labels:

- `needs-triage` — maintainer needs to evaluate
- `needs-info` — waiting on reporter for more information
- `ready-for-agent` — fully specified, ready for an AFK agent
- `ready-for-human` — needs human implementation
- `wontfix` — will not be actioned

Every triaged task should carry exactly one category label and one triage state label. If state labels conflict, flag it and ask the maintainer before doing anything else.

Use labels for triage state. Native Backlog statuses are only board columns: `todo`, `in-progress`, and `done`. Backlog.md is the issue tracker.

State transitions: a new task normally gets the `needs-triage` label first; from there it moves to `needs-info`, `ready-for-agent`, `ready-for-human`, or `wontfix`. `needs-info` returns to `needs-triage` once the reporter replies. If triage verifies the task is already fixed or already implemented, set native status to `done` without using `wontfix`. The maintainer can override at any time — flag transitions that look unusual and ask before proceeding.

## Backlog.md commands

Use the local `backlog` CLI:

- List tasks: `backlog task list --plain`
- Search: `backlog search "<query>" --plain`
- View task: `backlog task view <id> --plain`
- Create task: `backlog task create "<title>" -d "<description>" -s todo -l bug,needs-triage --plain`
- Set board status: `backlog task edit <id> -s todo --plain`, `-s in-progress`, or `-s done`
- Set category + triage labels: `backlog task edit <id> -l bug,ready-for-agent --plain` or `backlog task edit <id> -l enhancement,needs-info --plain`
- Set plan: `backlog task edit <id> --plan "<agent brief>" --plain`
- Set or append notes: `backlog task edit <id> --notes "<notes>" --plain` or `--append-notes`
- Reject work: label it `wontfix` and move to board status `done`: `backlog task edit <id> -l bug,wontfix -s done --plain`; archive only if the maintainer asks.
- Mark verified resolved work: move to board status `done` with `backlog task edit <id> -s done --plain`; keep the category label and do not add `wontfix`.

For multi-line text, prefer real newlines inside the quoted argument or repeated `--append-notes` calls. Avoid shell-specific newline escapes.

## Invocation

The maintainer invokes `/triage` and describes what they want in natural language. Interpret the request and act. Examples:

- "Show me anything that needs my attention"
- "Let's look at task-42"
- "Move task-42 to ready-for-agent"
- "What's ready for agents to pick up?"

## Show what needs attention

Query Backlog.md and present three buckets, oldest first:

1. **Missing category** — no `bug` or `enhancement` label.
2. **`needs-triage`** — evaluation in progress.
3. **`needs-info` with reporter activity since the last triage notes** — needs re-evaluation.

Show counts and a one-line summary per task. Let the maintainer pick.

## Triage a specific task

1. **Gather context.** Read the full task with `backlog task view <id> --plain`, including description, labels, status, plan, notes, acceptance criteria, dependencies, references, docs, and dates. Parse any prior triage notes so you don't re-ask resolved questions. Explore the codebase using the project's domain glossary, respecting ADRs in the area. Read `.out-of-scope/*.md` and surface any prior rejection that resembles this task.

2. **Recommend.** Tell the maintainer your category and state recommendation with reasoning, plus a brief codebase summary relevant to the task. Wait for direction.

3. **Reproduce (bugs only).** Before any grilling, attempt reproduction: read the reporter's steps, trace the relevant code, run tests or commands. Report what happened — successful repro with code path, failed repro, or insufficient detail (a strong `needs-info` signal). A confirmed repro makes a much stronger agent brief.

4. **Grill (if needed).** If the task needs fleshing out, run a `/grill-me` session.

5. **Apply the outcome:**
   - `ready-for-agent` — set category + `ready-for-agent` labels, keep board status `todo`, then write the agent brief to the task plan field ([agent-brief.md](reference/agent-brief.md)).
   - `ready-for-human` — set category + `ready-for-human` labels, keep board status `todo`, then write the same structure to the task plan field, but note why it can't be delegated (judgment calls, external access, design decisions, manual testing).
   - `needs-info` — set category + `needs-info` labels, keep board status `todo`, then write triage notes to implementation notes (template below).
   - `wontfix` (bug) — set labels to `bug,wontfix`, set board status to `done`, and write the explanation to final summary.
   - `wontfix` (enhancement) — write to `.out-of-scope/`, mention it in final summary, set labels to `enhancement,wontfix`, and set board status to `done` ([out-of-scope.md](reference/out-of-scope.md)).
   - Already fixed/implemented — only after evidence-backed verification that the bug is already fixed or the requested enhancement already exists. Set board status to `done`, keep the correct category label, do not add `wontfix`, and write a final summary with the evidence checked.
   - `needs-triage` — set category + `needs-triage` labels and keep board status `todo`. Optional notes if there's partial progress.

## Quick state override

If the maintainer says "move task-42 to ready-for-agent", trust them and apply the triage label directly. Confirm what you're about to do (label changes, board status changes, plan/notes/final-summary updates), then act. Skip grilling. If moving to `ready-for-agent` without a grilling session, ask whether they want to write an agent brief.

## Needs-info template

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

Capture everything resolved during grilling under "established so far" so the work isn't lost. Questions must be specific and actionable, not "please provide more info".

## Resuming a previous session

If prior triage notes exist on the task, read them, check whether the reporter has answered any outstanding questions, and present an updated picture before continuing. Don't re-ask resolved questions.
